# Phase 2: Core Dashboard UI Implementation

Phase 1 successfully laid the foundation, including database schema, API routes, mock simulator, and the main dashboard shell layout. Phase 2 focuses on building the data-dense, real-time UI components that populate the `DashboardShell`.

## Implementation Strategy

We will build the primary UI widgets utilizing the "Terminal Neon" design aesthetics defined in `globals.css`. We'll connect these panels to our `useAgents` and `useTasks` mock hooks.

### Task 1: Base UI Primitives
Build reusable styled components to ensure visual consistency across the dashboard.
- [ ] `Badge.tsx` — Status tags with correct neon colors (Working, Idle, Error).
- [ ] `ProgressBar.tsx` — CPU/Memory usage bars.

### Task 2: The Agent Fleet Grid
The cornerstone of the control panel, displaying all agents and their live hardware/task stats.
- [ ] Create `AgentCard.tsx` showing name, role, status dot, and resource usage.
- [ ] Create `AgentGrid.tsx` grouping agents by `serverId`.
- [ ] Integrate into `DashboardShell.tsx`.

### Task 3: Task Pipeline Kanban
A real-time view of active tasks categorized by their current status.
- [ ] Create `TaskCard.tsx` displaying priority, title, and assigned agent.
- [ ] Create `TaskPipeline.tsx` splitting tasks into columns: `Queued`, `Active`, `Done`.
- [ ] Integrate into `DashboardShell.tsx`.

### Task 4: Infrastructure & Metrics Panels
Visuals to monitor overall server health and throughput.
- [ ] Create `InfrastructurePanel.tsx` reading server stats from the mock engine.
- [ ] Create `PerformanceMetricsPanel.tsx` rendering simplified tabular summaries.

### Task 5: Alert Feed
A sweeping log of activities.
- [ ] Create `AlertFeed.tsx` listening to mocked socket alerts or generating random feed items.
- [ ] Ensure autoscroll behavior.

---
*Once these tasks are built, the dashboard will look and feel exactly like the 50" Command Center spec!*
